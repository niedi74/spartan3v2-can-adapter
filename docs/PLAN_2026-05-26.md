# Spartan + M5 — Plan und Stand 2026-05-26

## Soll-Architektur

```
123TUNE+ (BLE)  ──>  Spartan ESP32  ──>  M5 Dial (Gateway-Mode)
Spartan 3 v2 (CAN) ─>   Hub:                Direktmodus:
                       Spartan3-Hub          Scan 123\tune+
                       7f510001-...           NUS-Service
                       Notify 250 ms          Notify len=5
WiFi:  Spartan3-Setup (AP) + Z00-Station (STA)
```

Spartan = **zentrales Engine-Bay-Gateway**. M5 hat zwei Modi:

- **CONN_DIRECT_123**: M5 verbindet direkt mit 123TUNE+ (funktioniert solide)
- **CONN_SPARTAN_GATEWAY**: M5 verbindet mit Spartan3-Hub, bekommt Spartan + 123-Daten kombiniert

### P2 — Spartan->M5 Langzeit-Stabilitaet
- Hardening Patches sind aktiv (Notify-Return, Watchdog, Conn-Params)
- Neues Log alle 2s: `BLE hub: clients=N notify_ok=X notify_fail=Y streak=Z adv=1`
- Bei Fahrt 30+ min beobachten, ob notify_fail/streak nennenswert wachsen

### P3 — M5 UI-Lag im Gateway-Mode
- decodeGatewayPayload macht 5+ String-Allokationen pro Notify im BLE-Callback
- Ziel: sscanf auf C-Buffer, Flag-basierte Parse-Trennung Notify -> loop()

- BLE Bonding/Security erst nach stabilem Livebetrieb
- /log Ringbuffer Endpoint optional

## Bring-up Wartezustand (Hardware)

### CAN
- SN65HVD230 CAN-Transceiver Breakout bestellen
- Verdrahtung: 3V3, GND, GPIO17 (TX), GPIO16 (RX), Blau=CANH, Lila=CANL
- Spartan-Power 12V: Rot = +12V, Weiss/Schwarz = GND, 5A-Sicherung

### UART (fuer Spartan-Konfig)
- Pegelwandler oder Spannungsteiler fehlt noch
- Orange (Spartan TX, 5V) -> Spannungsteiler 10k/20k -> ESP GPIO 26 (RX)
- Gelb (Spartan RX) -> ESP GPIO 27 (TX) direkt
- Grau -> ESP GND
- Test-Vorlage: USB-UART-Konverter am Tisch hat schon funktioniert
